## 模块一：Context / Performance —— 200 轮长会话如何压缩

### 我在 LinAgent 里的实现

**超过 `maxMessages` 条数就把最老的一段折成一条 system 摘要**。默认 24 条阈值、保留最近 8 条。压缩用 LLM 走一遍，失败降级到启发式拼接。

跑到 200 轮真正会遇到的问题：

**按条数不按 token 是错的**。8 条消息塞几十 KB 的工具结果照样不触发，100 条纯"你好"倒会触发一次。真实系统一定要按 token 数。我在 README 里已经把这条列成了短板。

**一刀切折叠是错的**。用户目标句（该长期保留）、工具日志（该早扔）、错误记录（该保留但可以精简）都被同等对待。一次压缩下去，模型经常忘掉最开始用户要它做什么。

### 200 轮的正确做法（我会做的迭代版）

不是"压缩得更狠"，而是**把 context 拆成三层，各自有独立的保留策略**：

```
┌ 目标层（Working Memory）────────────────────┐
│ 用户当前会话的 goal / 约束 / 关键决定       │
│ 从不压缩，永远置顶                          │
└─────────────────────────────────────────────┘
┌ 摘要层（Compacted History）────────────────┐
│ 早期对话的分段摘要（每 20-30 轮一段）       │
│ 按时间衰减，最老的先合并 / 丢弃             │
└─────────────────────────────────────────────┘
┌ 现场层（Recent Turns）─────────────────────┐
│ 最近 8-15 轮完整消息（含工具调用与结果）    │
│ 到阈值就把最老那批折进摘要层                │
└─────────────────────────────────────────────┘
```

**核心不同点**：**目标层是单独维护的**。第一次触发压缩时，让 LLM 从头到当前，抽出一份"用户在这个会话里到底想做什么" —— 这份东西之后每轮都注入 system prompt，但**不参与压缩**。

用户 200 轮里可能改过一次目标（"其实我想的是 X 不是 Y"），目标层要能被覆盖 —— 每 30 轮左右重新抽一次，跟旧的对比，有差异就更新。

### 流畅度怎么保证

流畅度差不是"压缩太狠"造成的，是**关键实体丢失**造成的。用户 200 轮里提过的人名 / 项目名 / 文件路径 / 关键数字，摘要 LLM 经常把它们抽象成"一些实体"、"多个文件"，然后下轮问答就断了。

一个可操作的解法：**在摘要 prompt 里显式约束保留实体原文**：

> 生成摘要时：人名、项目名、文件路径、URL、代码符号、数字必须原文保留；只压缩描述性语句和过程。

这个约束比"简短摘要"重要得多。做完这个改动，我预计流畅度掉线的主要来源就没了。

### 为什么不做更复杂的（RAG / vectorize old turns）

对话历史 vectorize + RAG 检索是"看起来对"的方案。但代价：
1. 每次都要嵌入索引 + 相似度查询，增加 P99 延迟
2. 检索命中率取决于用户当前输入的 query 质量，对话里很多"接着上面继续"、"那样吧"是根本查不到的
3. 引入一层新的黑盒 —— 出问题不好 debug

摘要 + 目标层的组合是更简单的方案，覆盖 90% 场景。**在有真实数据证明覆盖不了之前，不上向量检索**。

## 模块二：Memory —— 半个月后重问过的问题怎么召回

### 先分辨"重问"是哪种

用户"问过"这件事至少有 3 种：
- (a) **完全一样的字面问题**（`我叫什么名字？`）
- (b) **同义不同表述**（上次问"我年纪多大"，这次问"我几岁了"）
- (c) **主题相同但角度不同**（上次问"我住哪"，这次问"我常去的咖啡店"—— 用户以为你会记得他所在城市）

三种召回难度不一样，用同一个方法都不合理。

### LinAgent 目前的做法 + 局限

我做的是**分层记忆**：identity / preferences / facts / ongoing 四层。检索用 Jaccard 词面相似度，加上"identity + preferences 永远注入，facts + ongoing 按关键词命中 top-K"。

这套对 (a) 挺好 —— 字面相似 Jaccard 就能命中。**对 (b) 就已经开始漏了**，Jaccard 需要词面重叠，"年纪多大" vs "几岁了" 一个词都不共。**对 (c) 完全没戏**，词面上根本不相关。

### 更合理的召回架构

不是换检索算法（换到 embedding 也只是 b 变好、c 依然差），而是**在事实抽取时就做好铺垫**：

**1. 抽取时归类到"层"，注入策略跟着层走**

我目前分了 4 层。真正关键的是**哪些永远注入、哪些按需注入**：

- **每次都注入**：用户的基础身份 + 偏好（identity / preferences 两层）—— 熟半个月后，这些不该等用户问才召回。用户说"我叫小林"，之后**任何**对话开头都该带上"用户叫小林"。这解决 (a) 和 (b) 里"我叫什么"这类问题 —— 根本不需要检索。

- **按语义相关性注入**（这里才需要检索）：facts / ongoing 两层。用户问"我常去的咖啡店"时，检索到"用户住杭州"、"用户喜欢喝咖啡"两条注入。

**2. 检索一定要有 embedding**

embedding 解决 (b)（同义不同表述）。这是 mem0 / Zep 都做了的事，也是我在 LinAgent README 里明确列的短板 —— 没做是因为不想引依赖，不是不该做。

**3. 主题级检索解决 (c)**

用户问"我常去的咖啡店"，字面上跟 memory 里"住杭州"零重叠。但如果每条 fact 都在存入时附带一个 topic 标签（`身份/住址`、`偏好/饮品`），检索时不只匹配 query 本身，也匹配 query 抽出来的 topic —— "咖啡店" 的 topic 包含 `饮品`、`地点`，就能命中"住杭州"和"喜欢咖啡"。

topic 抽取是抽取器 LLM 顺带做的事，成本可控。

**4. 只显式召回、不做"agent 主动查"**

我目前的做法是：**在 system prompt 拼装阶段静默注入**，agent 不感知有 memory 存在。**不做 tool-based 检索**（"agent 决定要不要调 recall 工具"）。

理由：让 agent 自己决定何时召回，等于把 recall 的准确率押在 agent 的判断力上，而且多消耗一次 LLM 调用。静默注入的坏处是每次都注入即使不需要 —— 但 identity/preferences 两层加起来一般 100-300 token，代价可以接受。

### 熟半个月这个场景的具体建议

半个月约 40-100 次会话，identity/preferences 早就抽出来了。用户重问已经问过的问题时：

- 如果是 (a) 或"我 XX"这类基础身份问题 → 依赖 identity 层每次注入，直接答，不用检索
- 如果是 (b) 或 (c) → embedding 检索 facts/ongoing 层
- **无论哪种，答完之后不要提"你之前告诉过我"**（除非用户本人问"你记得吗"）—— 用户体验上，"记得"是通过流畅接下去表现的，不是通过邀功表现的

## 模块三：Task —— 每天 9 点复盘

### 拆成 3 个子问题

题目看着简单但其实是三件事的组合：

1. **调度**（每天 9 点触发）—— 定时任务，纯工程问题
2. **数据源**（昨天聊了什么）—— 跨会话 memory 或 session 存档的读取
3. **产出**（复盘总结）—— agent 单次 turn 生成

单独看每一件都不难，架构关键是**它们之间怎么串起来**。

### 我会怎么设计

**放在服务端定时任务里**，不是让 agent"自己记住 9 点要做"—— 后者不可靠（会话可能根本没在 9 点被打开）。

```
Scheduler (cron)                                                   
    │                                                              
    ├─ 9:00 触发                                                   
    ▼                                                              
[起一个特殊 session]                                                
  system = "你是复盘助手。总结用户昨天的对话。"                    
  user = "开始今天的复盘。"                                        
    │                                                              
    ▼                                                              
Agent 内部循环 →  调 recap_yesterday 工具                          
                    │                                              
                    ├── 读昨天 sessions 表                          
                    ├── 抽取每个 session 的核心决定 / 待办 / 情绪  
                    └── 返回结构化摘要                             
    │                                                              
    ▼                                                              
Agent 综合 → 生成一段人可读的复盘                                  
    │                                                              
    ▼                                                              
写入 memory（"2026-XX-XX 复盘"作为 ongoing 层的一条 fact）         
    │                                                              
    ▼                                                              
推送到用户（IM / 邮件 / 首次打开 agent 时显示）                    
```

### 3 个关键的设计决定

**1. 复盘用独立 session，不复用用户的日常 session**

如果在用户当前对话里插一段"顺便复盘一下昨天"，会打乱当前上下文，用户体验很差。独立 session + 单独一次 LLM 生成，产出作为**外部通知**送达。

**2. `recap_yesterday` 是个显式工具，不是把昨天所有 session 塞进 context**

昨天可能有 20 段对话、几万 token。全塞进 context 又慢又贵。工具内部先做初筛：只挑有实质内容的 session（消息数 > N，或者含工具调用），每段先本地摘一遍，最后拼给 agent 综合。

**3. 复盘本身写回 memory，形成周-月复盘的基础**

日复盘写入 memory 后，下周做周复盘时 `recap_last_week` 工具就不用重新读 7 天 × 20 段对话，直接读 7 条日复盘就够了。**分层 recap**：日、周、月各是上一层的聚合。

### 长程任务里"忘掉目标"这个更普遍的问题


长程任务忘目标的**主流方案**：

- **目标置顶注入**（把目标写在 system prompt 或每轮第一条 user 消息）—— 简单、能解决 80%，但 system prompt 每变一次就击穿缓存
- **心跳提醒**（每 N 步 agent 自己重复目标）—— 用 token 换准确性
- **目标即工具**（`get_goal()` 工具让 agent 想起来时主动查）—— 让 agent 判断何时查，反而不稳
- **状态机 + checkpoint**（用户目标是状态机的一个 slot，永远存在，不参与压缩）—— 我在模块一答的"三层 context"就是这个

我更倾向状态机 + checkpoint 的方案，因为它同时解决了长会话压缩里的"目标丢失"和长程任务里的"跑偏"—— 是同一件事。

## 模块四：Tool / Session Runtime —— 异步工具 + busy 状态收消息

题目里的两问其实是一个 runtime 状态机的两面，我合成一题答。

### 异步工具的核心矛盾

同步工具：调 → 阻塞 → 返回。语义简单，但工具跑 30 秒用户就干瞪 30 秒。
异步工具：调 → 立刻返回一个 handle → 事件回来时通知。**语义不匹配 LLM 的循环模型**（LLM 每轮只能"消化"当前上下文，异步事件本质是外部注入）。

关键设计问题：**agent 循环、异步事件、用户新消息**这三个东西的关系。

### 我会怎么设计

**Session 有一个显式的状态字段**：

```
   ┌──────────┐        用户输入 / 定时触发
   │   idle   │◀────────────────────────┐
   └────┬─────┘                         │
        │ 开始 chat                     │
        ▼                               │
   ┌──────────┐                         │
   │   busy   │─── agent 循环运行中 ────┤
   └────┬─────┘                         │
        │ 需要调异步工具                │
        ▼                               │
   ┌──────────────┐   工具完成事件到达  │
   │ awaiting_    │──────────────────►┐ │
   │ async_tool   │                   │ │
   └──────────────┘                   │ │
        △                             │ │
        │  用户在这时发新消息          │ │
        ▼                             ▼ │
   ┌──────────────┐              ┌───────────┐
   │ 消息进入队列 │              │ 恢复 busy │
   │ 不立刻处理   │              │ 处理事件  │
   └──────────────┘              └─────┬─────┘
                                       ▼
                                 处理完队列消息 → idle
```

**关键规则**：

**1. `busy` 或 `awaiting_async_tool` 时**：
- 用户新消息 → 追加到 session 的**消息队列**，不立刻塞进 agent 的 context
- UI 层给用户一个明确的"agent 正在处理中，你的消息会稍后送达"反馈
- **不是丢弃、不是打断、也不是立刻塞进当前 turn** —— 后者会污染 agent 当前正在做的事

**2. 异步工具完成事件到达时**：
- 如果 session 是 `awaiting_async_tool` 且工具 id 匹配 → 事件作为 `tool_result` 塞进 context，恢复 agent 循环
- 如果 session 已经 `idle`（用户已经放弃这个任务开了新话题） → 事件写入一条"外部通知"，不打断当前对话，agent 下一次 turn 开头可选择性提及

**3. 消息队列清空时机**：
- Agent 循环走到 `final_answer` → 进 `idle` → 处理队列 → 如果有堆积的用户消息，把它们合并成一条（或按顺序处理），启动下一次 chat

### 用户体验层面的额外设计

**"打断"要显式提供**。有时候用户就是想中断当前 agent 循环、立刻处理新消息。给一个 `/interrupt` 或 UI 按钮，语义是"取消当前 turn，把当前进度作为部分结果保留"—— 而不是让用户猜"我发消息 agent 会不会打断"。

**长时间 busy 有主动汇报**。异步工具跑 > 30 秒时，让 runtime 自动往 UI 推一条"仍在等待 X 工具完成"—— 不是让 agent 生成，是 runtime 层直接推。用户不会因为看不到反馈以为程序挂了。

### 为什么不做"用户消息直接注入 running turn"

看起来很酷（"边跑边收新指令"），但 LLM 的循环模型不支持中途改 context —— 强插消息会导致：agent 已经 plan 好的下几步失效、tool_use 和 tool_result 的配对错位、模型看到不连贯的历史。

**正确的做法是把"用户想改主意"表达成一次 interrupt**，而不是伪装成"新消息"。语义清晰的中断信号比隐式的消息注入更可靠。

## 模块五：Runtime 架构对比 —— Claude Code 与国内 function calling

我在 LinAgent 里就用了自定义 JSON 协议（既不是 Claude 的 tool_use content block、也不是 OpenAI 的 tool_calls），所以这题三种做法我都碰过。

### 三种做法的本质

**OpenAI-compatible function calling（国内绝大多数厂商都跟这个）**：

工具调用是**消息级独立字段**，跟 content 分开。

```json
{
  "role": "assistant",
  "content": null,
  "tool_calls": [{"id":"call_1","function":{"name":"...","arguments":"..."}}]
}
```

工具结果是**独立的 role="tool"消息**。arguments 是 stringified JSON（历史包袱，每次要 `JSON.parse`）。

**Claude 的 content block 模型**：

工具调用是**content 数组里的一个块**，跟文本、思考混编。

```json
{
  "role": "assistant",
  "content": [
    {"type":"thinking","text":"..."},
    {"type":"text","text":"让我查一下"},
    {"type":"tool_use","id":"toolu_1","name":"...","input":{...}}
  ]
}
```

工具结果是 user 消息里的 `tool_result` 块。

**Claude Code 更进一步**：某些工具（比如 bash）的输出**是流式的**，边跑边推 —— 从最初的"一次性文本 tool_result"演化成"tool_result 可分段"。

### 优缺点对比

**OpenAI function calling 的优点**：

- **生态铺得广**。所有兼容 OpenAI 的模型都支持这套（DeepSeek、Kimi、千问、GLM、vLLM、Ollama）。用一套代码打十个 provider
- **结构简单**：content 是字符串，tool_calls 是独立字段，schema 一眼看清
- **`strict: true` 提供 grammar-constrained decoding**，物理保证 tool_calls 格式合法

**缺点**：

- **arguments 是 stringified JSON**，每次要 parse。这是 v1（chat completion API）遗留，OpenAI 自己在新的 responses API 里改成对象了，但**国内厂商还都停留在老 API**
- **消息级独立字段**决定了 tool_calls 和 text 是**顺序不可控**的：只能"先说完话再调工具"或"直接调工具"，不能"说一半话，调个工具，看结果继续说"—— 这在需要边说边查的复杂 agent 里很憋屈
- **thinking 只能塞进 content 字符串里**（`<thinking>...</thinking>` 这种老 hack），跟 tool_calls 一样是消息级独立字段，没法自然交错

**Claude content block 的优点**：

- **块级顺序自由**：可以"想 → 说 → 调工具 → 拿结果 → 再想 → 再说" 全都在**一条消息里**表达。这对 chain-of-thought + tool use 交错的场景是**结构性优势**
- **加新能力只加新块类型**：thinking / image / document / tool_use / tool_result 都是 block type。不像 OpenAI 每加一个功能就要加一个顶层字段（tool_calls / refusal / function_call...）
- **claude code 的流式 tool_result** 建立在这套之上很自然 —— 就是 block 分段推送

**缺点**：

- **生态窄**。只有 Anthropic 家 + AWS Bedrock / Google Vertex 上的 Claude 支持
- **代码写起来更啰嗦**：不能 `msg.content.includes("hello")`，要先遍历 blocks 找出 `type === "text"` 的
- **国内厂商没跟**，写跨 provider 代码要写两套

### Claude Code 流式工具输出的更深层设计

题目问的是"工具输出方式"。Claude Code 相比国内 function calling 最不一样的一点：

**bash 工具的输出是流式的**。你在终端里跑 `npm install` 会看到日志一行行往下滚 —— 这不是前端在轮询，是 API 层就支持工具输出**分块推送**。

普通 OpenAI 兼容的 tool_result 一定是**一次性**回给 LLM：工具跑完 → 一整段 result → 塞给模型。跑 5 分钟的 bash 命令，用户在这 5 分钟里干瞪着，除非前端自己搞个 WebSocket 单独推日志。

Claude Code 的做法：**工具执行边跑边把日志作为 `tool_result` 块的增量推给模型**，模型看到进度就能实时反应（"看起来还在下载依赖，我等一下"、"发现 npm registry 慢，切个源试试"）。这是 UX 上的巨大改善。

**代价**：工具端要实现流式接口，agent runtime 要支持增量 tool_result。工程复杂度上一个台阶。

### 为什么我在 LinAgent 里选自定义 JSON 协议

**跨 provider 通用 > grammar-constrained decoding**。我要用 DeepSeek 跑 demo（免费）+ 允许换任何 provider，就必须走两家协议都支持的最小公倍数 —— 字符串 content 里塞 JSON。

**代价**：丢了 `strict: true` 保证，所以要写 `parser.ts` 的 balanced-brace 扫描 + 错误回喂来兜底。工具描述都在 system prompt 里显式列出，每轮多花 ~1400 token（README 里坦白列了）。

如果我做的是生产系统而不是 demo，路径应该是：**默认走原生 tool_use / tool_calls，只在跑非官方 provider 时降级到 JSON 协议**。这是 v3 的方向 —— 我 README 里也写了。

### 总结

- **OpenAI 兼容**：赢在生态，输在表达力
- **Claude content block**：赢在表达力，输在生态
- **Claude Code 的流式工具**：把 UX 从"命令行等半天"变成"看进度实时反应"，代价是工程复杂度
- **自定义 JSON**：跨 provider 通用性最好，代价是没有 grammar 保证 + 多花 token

选哪种取决于你的**约束条件**是什么 —— 生产系统一般用原生协议、跨 provider 工具库常用自定义 JSON。没有绝对的对错。
